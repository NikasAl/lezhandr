### 1.2 ✅ Дублирование паттернов диалогов — Исправлено

**Проблема была решена:** Диалоги вынесены в отдельные функции с унифицированным API. Каждый диалог теперь имеет чёткий контракт:
- Принимает `BuildContext`, `WidgetRef`, и специфичные параметры
- Возвращает результат операции
- callback для рекурсивных вызовов (например, `onQuestionUpdated`)

**Оставшиеся улучшения (опционально):**
При добавлении новых типов артефактов можно создать базовый класс/миксин для диалогов.

### 1.3 🟡 `solution_detail_screen.dart` — Встроенные виджеты

**Проблема:**
В файле определены 8 классов:
- `_StatusCard`
- `_StatChip`
- `_SolutionTextSection`
- `_EpiphaniesSection`
- `_EpiphanyItem`
- `_QuestionsSection`
- `_QuestionItem`
- `_HintsSection`

**Почему это проблема для ИИ-агента:**
- При работе с одним виджетом видны все остальные
- Сложно найти нужный участок кода
- Риск случайных изменений в соседних виджетах

**Рекомендация:**
Вынести в отдельные файлы или хотя бы в `_components/` подпапку:
```
lib/presentation/screens/solutions/detail/
├── status_card.dart
├── solution_text_section.dart
├── artifacts_sections.dart  // или три отдельных файла
```

**Оценка усилий:** 1-2 часа
**Выгода для ИИ-агента:** Средняя

## 2. Участки с умеренным приоритетом

### 2.1 🟡 Отсутствие централизованной обработки ошибок

**Проблема:**
Обработка ошибок разбросана по экранам:
```dart
// problem_detail_screen.dart
if (result.error != null) {
  if (mounted) {
    ScaffoldMessenger.of(context).showSnackBar(
      SnackBar(content: Text('Ошибка OCR: ${result.error}')),
    );
  }
}

// solution_session_screen.dart
if (hint == null || hint.id == null) {
  if (mounted) {
    ScaffoldMessenger.of(context).showSnackBar(
      const SnackBar(content: Text('Не удалось создать запрос подсказки')),
    );
  }
}
```

**Рекомендация:**
Создать утилиту для показа ошибок:
```dart
class ErrorHelper {
  static void show(BuildContext context, String message, {String? title}) {
    if (!context.mounted) return;
    ScaffoldMessenger.of(context).showSnackBar(...);
  }

  static void showApiError(BuildContext context, ApiException e) {
    show(context, e.message);
  }
}
```

**Оценка усилий:** 1 час
**Выгода для ИИ-агента:** Средняя — меньше дублирования

### 2.2 🟡 Магические строки и константы

**Проблема:**
Строковые литералы в коде:
```dart
status: 'completed'  // должно быть SolutionStatus.completed.name
path.contains('/auth/device-register')
Text('Озарение')
Text('Подсказка')
```

**Рекомендация:**
- Использовать enum вместо строк где возможно
- Создать `AppStrings` для UI-текста (опционально для локализации)
- Создать `ApiEndpoints` для путей API

**Оценка усилий:** 1-2 часа
**Выгода для ИИ-агента:** Низкая, но улучшает поддерживаемость

## 5. Вывод

**Принципиальная разница есть?** Да, но не критическая.

Проект имеет хорошую архитектурную основу:
- Чистое разделение слоёв (data / presentation / core)
- Riverpod для state management
- Типизированные модели
- Абстракция API через репозитории

**Основная проблема** — несколько крупных файлов с дублированием паттернов. Это создаёт риск для ИИ-агента:
- Труднее понять контекст
- Выше вероятность случайных изменений
- Дублирование логики вместо переиспользования

**Рекомендация:**
Если планируется активная доработка проекта через ИИ-агента, стоит выполнить **пункт 1** (расщепление solution_session_screen.dart). Остальные улучшения можно делать по мере необходимости.
